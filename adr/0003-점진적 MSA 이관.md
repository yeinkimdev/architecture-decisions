# ADR-0003: DB는 그대로 두고 API만 재설계하는 점진적 MSA 이관

- Status: Accepted (진행 중)
- Date: 2022-XX-XX  ← 본인 시작 시점으로
- Deciders: 김예인
- Tags: #migration #msa #legacy #strangler-fig

## Context
- 사내 SaaS 한 축이 십여 년 운영된 PHP 모놀리스 위에 있었음. 단일 Apache + PHP + 단일 Oracle + 단종된 검색엔진 구성. 
  여러명이 거쳐간 코드 사본이 여러 갈래로 남아있고, 신규 기능 추가할 때 영향 범위가 추적이 안 되는 상태였음.

- 처음 구상은 DB랑 API 다 새로 설계하는 방향이었음. 도메인을 새로 그리고, PK/FK를 다시 잡고, 컬럼명도 정리하고, REST API 만들고. 이걸 끝내고 백오피스까지 새 시스템으로 한 번에 넘어가는 청사진이었음.

- 그런데 실제로 까보니 **운영 DB의 테이블이 3000개가 넘었음**. 코드에서 쓰는 거 외에 배치/리포팅용/임시 테이블/이력 테이블이 누적된 결과.
  도메인 분류부터 매핑·정합성 검증·이관·역방향 호환까지 따져보니 DB 마이그레이션 작업만 최소 1년 이상이 예상됐고, 그동안 신규 기능은  레거시 위에서 계속 만들어져야 했음.

- 즉, DB설계를 새로 하는 동안 비즈니스가 멈추지 않는다가 사실상 불가능했음. DB 재설계를 끝까지 밀어붙이면 신규 시스템이 나오기도 전에 방향성이 무너질 위험이 컸음.

## Decision
1. **DB는 레거시 스키마를 그대로 유지하고, API 계층만 재설계한다.**
   - 신규 서비스는 레거시 Oracle을 그대로 읽고 쓰되, 도메인 단위로 리포지토리/매퍼를 끊어서 사용한다.
   - 컬럼명/스키마 이상한 부분은 신규 서비스의 도메인 객체로 흡수 (Anti-Corruption Layer 성격).
2. **Strangler Fig 패턴으로 도메인을 하나씩 흡수한다.** 한 도메인을 옮긴 뒤에는 최소 한 달 양쪽을 운영하면서 결함을 잡고, 안정되면 레거시 라우트를 닫는다.
3. **빌드 순서는 인프라 → 고객 서비스 → 백오피스** 순으로 잡는다.
   - DB 재설계가 빠진 상태에서 백오피스(데이터를 직접 보는 화면)는 큰 가치를 만들기 어려움 → 고객 영향이 큰 도메인부터.
4. 도메인 경계는 "변경 빈도가 함께 가는 묶음"과 "데이터 일관성 경계"를 기준으로 그린다. 단, 처음부터 완벽하지 않다는 것을 전제로 두고 두 번 옮길 수 있게 가벼운 단위로 분리.

## Decision Timeline (실제 진행 흐름)
> ADR 정통 형식은 아니지만, 다년간 누적된 결정을 추적하기 위해 시간축으로 남김. 각 단계는 그 시점의 새 요구/제약이 트리거.

### Step 1 — Gateway 신설 (가장 먼저)
- 신규/레거시 라우팅의 분기점이 필요. 신규 도메인 서비스가 준비될 때마다 그 라우트만 신규로 옮길 수 있어야 함.
- Spring Cloud Gateway 채택. 인증/인가 필터도 일단 Gateway에서 처리 (이후 IdP 분리 단계에서 책임이 일부 떨어져나감 — Step 7 참조).

### Step 2 — Eureka 도입
- 신규 서비스가 늘어날 것을 전제로 서비스 디스커버리 필요. 당시 배포는 Docker 기반이라 호스트/포트 직접 박는 구조가 곤란.
- Spring Cloud Netflix Eureka 채택. Gateway가 Eureka에서 서비스 위치를 찾도록 구성.

### Step 3 — 세무 도메인 신설 (첫 번째 도메인 서비스)
- 가장 사용량이 많고, 레거시 코드 변경이 잦은 도메인이자 핵심 도메인
- 레거시 Oracle을 그대로 사용. 도메인 모델은 신규 서비스 안에서 다시 정의(컬럼명 불일치는 매퍼에서 흡수).

### Step 4 — 회원 도메인 신설
- 회원/조직 도메인 분리. 세무 도메인과 양방향 의존이 생기지 않도록 회원 쪽은 OpenFeign으로만 호출되는 방향으로 정리.

### Step 5 — 백오피스는 후순위로
- 원래 계획은 et4xkorea → membership → backoffice 였음.
- 그런데 DB 재설계를 하지 않은 상태에서 백오피스를 새로 만드는 건 결국 같은 데이터를 다른 화면으로 보는 것일 뿐, 이관의 가치가 작음.
- 또 백오피스는 운영팀이 일상적으로 쓰기 때문에 화면 변경 비용도 큼. **고객 서비스부터 안정화한 뒤에 옮기는 것으로 순서 조정.**

### Step 6 — 결제 도메인 신설 (요구사항 변화)
- 비즈니스 요구가 일회성 → **구독 결제**로 전환되면서 결제 도메인의 복잡도가 급증.
- 세무 도메인 안에 결제 로직을 두면 도메인이 비대해지고, 결제는 본질적으로 멱등성/상태 머신/감사 로그가 필요한 별도 책임이라 분리.
- **결제 도메인 신설.** 결제 상태 변경은 payment가 단독 소유, 다른 서비스는 이벤트나 API로만 관여.

### Step 7 — Docker → Kubernetes 전환, Eureka 제거
- 배포 환경을 Docker에서 Kubernetes로 옮김 (롤링/카나리/HPA 같은 운영 이점 + GitOps).
- K8s가 자체적으로 Service Discovery를 제공(ClusterIP + DNS), Eureka가 중복 책임이 됨.
- **Eureka 제거.** Gateway/내부 서비스 간 호출은 K8s Service 이름으로 대체. Spring Cloud LoadBalancer는 K8s Service 뒤에 위임하는 형태로 정리.

### Step 8 — SSO 요구 → IdP 분리, 인증 일부 분리
- 사내 SaaS 라인업이 늘어나면서 **단일 로그인(SSO)** 요구가 들어옴.
  Gateway 안에 인증을 두면 다른 서비스(라인업)와 인증 자산을 공유할 수 없음.
- OAuth 2.0 도입 결정. 인증서버(IdP)를 별도 서비스로 분리.
- 결과적으로 **인증 책임은 IdP, 인가 책임은 Gateway**로 나뉘었음.
  Gateway는 JWT 검증 + 권한 필터만 담당, 토큰 발급/사용자 검증은 IdP가 단독 책임.
- 이 단계는 그 자체가 큰 결정이라 별도 ADR(예정)로 분리해서 자세히 남길 예정.

## Alternatives Considered
- **A안 (선택됨): DB는 유지, API만 재설계 + Strangler Fig 점진 이관**
  - 장점: 다운타임 0, 위험 시간 분산, 옮긴 도메인부터 가치 발생, 중간에 멈춰도 운영 가능
  - 단점: 신규/레거시 양쪽 동시 운영 비용, 어댑터/매퍼 코드가 길게 살아남음, DB 부채는 그대로 남음

- **B안: DB + API 풀 재설계 (Big Bang)**
  - 장점: 끝나면 DB 부채까지 같이 청산
  - 단점: 테이블 3000개 마이그레이션이 1년+ 소요, 그동안 비즈니스가 멈춤. 출시 시점에 결함이 한꺼번에 터지는 구조라서 현실적으로 위험 부담이 컸음.

- **C안: 레거시를 그대로 두고 Refactor in Place**
  - 장점: 시스템 변경 없이 코드만 정리
  - 단점: 검색엔진(K2) 단종으로 어차피 이관 불가피. PHP 위에서 리팩토링해도 운영 환경/관측성/배포 자동화 측면 개선이 없음. 1~2년 뒤 같은 문제 반복.

## Consequences
- ✅ 다운타임 없이 도메인별로 점진 흡수 진행. 큰 사고 없이 9개 마이크로 서비스 단계적 완성
- ✅ 한 도메인을 옮길 때마다 신규 시스템의 이점(K8s 운영, GitOps, 관측성)을 바로 누릴 수 있었음. "다 끝나야 좋아진다"가 아니라 "옮기는 만큼 좋아진다"는 구조.
- ✅ 요구 변화(구독 결제, SSO, K8s 전환)에 따라 서비스를 새로 만들거나 제거하는 결정이 가벼웠음. 모놀리스에서는 사실상 못 했을 변화.
- ⚠️ **DB 부채는 남아있음.** 신규 서비스도 결국 레거시 Oracle을 보고있고, 컬럼명/관계 이상한 부분은 매퍼에서 흡수만 했지 해결한 건 아님. 추후 도메인별로 신규 DB로 옮기는 작업이 별도 ADR로 필요.
- ⚠️ 신규/레거시 양쪽 동시 운영 비용이 작지 않음. 모니터링·배포·장애 대응이 두 곳, 같은 도메인 수정이 양쪽에 들어가는 시기가 반드시 있음.
- ⚠️ 어댑터/매퍼 코드가 "임시"라고 만들어놓고 영구 코드가 되는 경향. 특히 레거시 DB를 보는 어댑터 — 이게 잘 안 사라짐.
- 🔮 재검토 트리거:
  - 도메인 단위로 신규 DB로 옮길 필요가 생기거나
  - 어댑터 코드가 도메인 코드보다 복잡해지거나
  - 양쪽 동시 운영 비용이 신규 단독 운영 비용을 넘어선다고 판단되면
  → 도메인별 DB 분리 ADR, 또는 잔여 레거시 EOL ADR을 새로 작성.

## Lessons Learned
- 처음에는 DB까지 같이 재설계하는 그림이 더 매력적으로 보였음. 깔끔하니까.
  근데 테이블 갯수를 실측해보고 나서 **"기술적으로 매력적인 결정"과 "운영 가능한 결정"이 다르다는 점**을 처음 체감함. 비즈니스를 멈출 수 없다는 제약이 모든 설계의 상한선이 된다는 걸 그때 배움.

- 도메인 경계는 처음에 완벽하게 못 잡음. 머리로 그린 경계는 실제로 서비스가 돌아가봐야 맞는지 알 수 있음. 그래서 처음부터 완벽한 경계를 그리려고 너무 오래 회의하는 것보다 **"틀려도 다시 옮길 수 있다"는 전제로 빠르게 옮기고 운영해보는 게** 결과적으로 더 빨랐음.

- 백오피스를 후순위로 미룬 결정도 같은 맥락. **DB 재설계를 안 한 상태에서 백오피스를 옮기는 건 같은 데이터를 다른 화면으로 보는 것에 가까웠음.** "옮기는 행위 자체"보다 "옮겨서 만들어내는 가치"를 기준으로 순서를 잡는 게 맞다는 걸 그때 정리함.

- Eureka가 1~2년 뒤 K8s 이관 때 제거된 것도 회고할 부분. 그 시점에는 올바른 선택이었지만, **인프라가 바뀌면 이전 결정도 다시 평가해야 한다**는 걸 보여주는 사례. "한 번 정한 건 계속 간다"는 가정이 MSA에서는 더 빨리 깨짐.

- 결제(payment)와 인증(IdP)이 둘 다 **요구 변화로 새로 분리됐는데**, 둘 다 처음 설계에는 없던 서비스였음. 처음부터 다 그리려 하지 않고, 필요해질 때 도메인을 분리할 수 있는 구조를 유지한 게 결과적으로 잘 작동함. (Strangler Fig의 부수 효과로, 신규 도메인을 추가하는 비용이 낮아진 셈)

- **신규/레거시 동시 운영 비용을 과소평가했음.** 모니터링·배포·장애 대응이 실질적으로 두 배가 되고, 양쪽이 다 살아있는 동안 도메인 변경이 들어오면 두 곳에 같은 수정을 해야 함. 이 비용 때문에 후속 도메인의 이관 속도를 의식적으로 빠르게 가져가야 했음.

# ADR-0005: 권한 어휘의 단일 진실 — 구독 단위 (PlatformRole)

- Status: Proposed
- Date: 2026-05-19
- Deciders: 김예인
- Tags: #authorization #domain-language #platform-role

## Context
- ADR-0004 에서 인가를 각각의 자원 서비스로 위임했음. 그러면 다음 질문은 — 자원 서비스가 어떤 어휘로 인가 결정을 내릴지.
- 사내 SaaS 라인업이 한 개가 아닌 여럿이라는 점이 권한 어휘에 영향을 줌. 회사가 세무/회계 도메인 안에서 여러 플랫폼을 운영하고 있고, 같은 회원 풀이 그 플랫폼들에 걸쳐 구독 단위로 접근함. 즉 회원이 어떤 플랫폼을 구독했는지가 곧 어떤 자원에 접근 가능한지를 결정함.
- 그런데 코드 베이스에 historical 으로 박혀있는 권한 어휘는 등급 (`ROLE_USER` / `ROLE_MEMBER` / `ROLE_ADMIN`) 임. 등급은 회원이 인증되었다는 정도만 표현하고, 어떤 플랫폼의 회원인지는 표현 못 함. 멀티 SaaS 환경에서 등급 어휘로는 인가 정책을 표현할 도구가 없음.
- 미래에 권한 모델은 더 세분화될 예정 — 무료 체험 / 베이직 / 프리미엄 / ... 이걸 등급이라는 일직선 모델로 표현하면 또 부족해질 것 같음. 등급 모델은 이전에 한 번 깨져서 별도 role 테이블이 사라진 적도 있음 (Journey Step 1/2).
- ADR-0004 의 무력화 사고에서 또 하나 드러난 점은, 권한 어휘가 코드 곳곳에 String 상수로 분산돼 있어서 한쪽이 바뀌어도 다른 쪽이 못 따라온다는 사실. 이번 ADR 은 어휘를 어디에 어떻게 묶을지까지 같이 정한다.

## Journey (권한 어휘의 변천)

### Step 1 — user / member / role 분리 시도 (별도 role 테이블 운영)
- ADR-0004 Journey Step 4 와 같은 시기. 외부 사용자(`user`) 와 내부 사용자(`member` = 운영자) 를 role 로 구분하려는 의도였음.
- DB 에 별도 role 테이블을 두고 회원과 N:M 매핑. 행 예: `ROLE_USER`, `ROLE_MEMBER`, `ROLE_ADMIN`.
- 이 구조의 한계가 곧 드러남 — `ROLE_MEMBER` 가 어떤 SaaS 의 멤버인지 표현 못 함. 회사가 SaaS 한 개일 때만 유효한 모델이었던 셈.
- 운영자(내부) 와 회원(외부) 을 같은 테이블에 두려던 시도도 어색했음. 인증 출처가 LDAP 과 자체 회원가입으로 다른데 한 테이블로 묶이니까 의미 없는 컬럼이 한쪽에만 채워지는 모델이 됨. 이 어색함이 ADR-0006 의 분리 결정으로 이어짐.

### Step 2 — IDP 분리 + role 테이블 폐기
- ADR-0004 Journey Step 5 의 IDP 분리 시점. IDP 가 인증 책임을 가져가면서 role 테이블의 의미가 약해졌음. 이제는 토큰 자체가 권한을 들고 있어야 했음.
- 일반 회원 인증 시 IDP 가 회원 서비스에 "이 회원이 어떤 플랫폼을 구독했는지" 를 물어보고, 그 결과를 토큰의 `roles` 클레임에 담음.
- 이때 어휘가 자연스럽게 진화함 — 등급(`ROLE_MEMBER`) 에서 구독(`ROLE_플랫폼명`) 으로. "회원 여부" 는 토큰 존재 자체로 표현되니까 등급 어휘는 더 이상 필요하지 않게 됨.
- DB 의 별도 role 테이블 폐기. 권한의 단일 진실은 회원 서비스의 구독 정보로 단일화.

### Step 3 — 어휘 정의의 분산 (현재까지의 상태)
- 진화한 어휘가 단일 정의 없이 코드 베이스 곳곳에 흩어져 있는 상태:
  - **회원서비스** — `PlatformRole` enum 정의. 자체 도메인 코드.
  - **IDP** — 회원의 enum 을 Feign 으로 받아 `ROLE_` 접두사 붙여 토큰의 `roles` 클레임에 넣음. enum 자체를 import 하지 않고 String 가공.
  - **Gateway** — ADR-0004 까지 자기 인가 정책에 등급 어휘(`ROLE_MEMBER`, `ROLE_ADMIN`) 를 박아둔 상태로 운영. 다른 컴포넌트는 진화했는데 Gateway 만 그대로 남아있었던 것. (이게 문제)
  - **자원 서비스** — 어휘를 모름 (Gateway 가 인가했으니까).
- 이 상태에서 한쪽이 갱신 안 되면 다른 쪽이 못 따라옴. ADR-0004 의 무력화 사고가 정확히 이렇게 발생.

### Step 4 — ADR-0004 의 후속 결정
- ADR-0004 가 인가를 자원으로 위임했으므로, 자원 서비스가 어휘를 직접 해석하게 됐음. 그러면 자원이 공유할 수 있는 단일 어휘 정의가 필요해짐.
- 사내 공통 라이브러리(ADR-0001) 가 그 자리에 자연스럽게 들어맞음 — 모든 자원 서비스가 의존하는 라이브러리. 여기에 `PlatformRole` enum 을 두면 자원 서비스가 같은 enum 을 import 해서 인가 정책을 작성할 수 있음.
- enum 으로 묶으면 컴파일러가 동기화를 강제. IDP 가 새 플랫폼을 추가하려면 enum 에 값을 먼저 추가해야 토큰을 발급할 수 있고, 자원 서비스도 같은 enum 을 사용하므로 자동으로 반영됨.

## Decision
1. 권한 어휘는 구독 단위(`PlatformRole`) 로 일원화한다. 등급 어휘는 사용하지 않는다. "회원 여부" 는 토큰 존재 자체로 표현한다.
2. `PlatformRole` enum 은 사내 공통 라이브러리에서 단일 정의한다. 회원 서비스에 있던 정의를 공통 라이브러리로 승격하고, IDP / 자원 서비스 / 회원 서비스 가 모두 같은 enum 을 참조한다.
3. IDP 는 인증 시 회원의 구독 정보를 받아 `PlatformRole.name()` 에 `ROLE_` 접두사 붙여 토큰의 `roles` 클레임에 담는다. Spring Security 의 `GrantedAuthority` 컨벤션 준수.
4. 자원 서비스는 토큰의 `roles` 클레임을 (Gateway 가 변환한 `X-Member-Authorities` 헤더로) 받아 자기 인가 정책에 사용한다. SecurityConfig 의 `hasAuthority(플랫폼명)` 또는 어노테이션 기반.
5. 어휘 일관성을 컴파일러로 강제하기 위해 도메인 어노테이션 `@RequirePlatform(PlatformRole.플랫폼명)` 을 도입한다. Spring Security 의 `@PreAuthorize` 위에 얹는 메타 어노테이션. 컨트롤러 메서드에 한 줄로 가드를 표현하고, enum 값을 직접 받으므로 오타나 잘못된 값이 컴파일 단계에서 잡힘.
6. 미래 등급(체험/베이직/프리미엄) 추가는 두 가지 옵션 중 하나로 처리하되, 결정은 그 시점에 별도 ADR 로 미룬다.
   - 옵션 1: PlatformRole 의 enum 값을 늘림 — `ROLE_플랫폼명_등급`. enum 폭증 위험.
   - 옵션 2: PlatformRole 과 평행한 `PlatformTier` enum 도입. 차원이 두 개로 분리. 인가 정책이 두 차원을 같이 봐야 함.

## Alternatives Considered
- **A안: 등급 + 구독 두 차원 모델 유지**
  - 등급(`ROLE_MEMBER`) 으로 회원 여부 판단, 구독(`ROLE_플랫폼명`) 으로 플랫폼 접근 여부 판단.
  - 장점: "회원이긴 한데 구독은 안 한 상태" 를 명시적으로 표현.
  - 단점: 어휘가 두 차원이라 인가 정책이 항상 두 가지를 같이 봐야 함. "구독했으면 회원" 이라는 자명한 관계를 매번 코드에 적어야 함. Step 1 에서 시도하다 깨진 모델.
- **B안: 단일 어휘를 OAuth2 표준 scope 로 표현**
  - `roles` 클레임 대신 `scope` 클레임에 `플랫폼:read 플랫폼:write` 같은 어휘.
  - 장점: OAuth2 표준에 가까움. 외부 통합 시 익숙.
  - 단점: scope 의 본래 의미는 "이 토큰으로 무엇을 할 수 있는가" 인데, 우리 도메인에서는 "회원이 무엇을 구독했는가" 라서 두 의미가 매번 일치하진 않음. 외부 표준 어휘를 들여오느라 도메인 의미가 흐려질 수 있음.
- **C안 (선택됨): 단일 차원 PlatformRole**
  - 회원 여부 = 토큰 존재. 구독 여부 = roles 클레임.
  - 장점: 어휘가 단순. 인가 정책이 "이 플랫폼 구독했는가" 만 보면 됨. 도메인 어휘(플랫폼) 가 코드 어휘에 그대로 노출.
  - 단점: 회원이긴 한데 구독은 안 한 상태가 빈 권한 배열로 표현되므로, 그 상태에서 접근 가능한 자원이 있다면 별도 처리 필요. 현실적으로는 그런 자원이 거의 모두 공개 경로라 큰 문제는 아님.

## Consequences
- ✅ 권한 어휘가 단일. 모든 자원 서비스 / IDP / Gateway 가 같은 enum 을 참조한다.
- ✅ 권한 추가/변경 시 enum 한 곳에 정의를 추가하고 자원 서비스 가드만 손보면 끝. IDP 매핑은 enum 값 그대로라 자동.
- ✅ ADR-0004 식 어휘 어긋남 사고가 구조적으로 발생하기 어려움. 컴파일러가 동기화를 강제해주니까.
- ✅ `@RequirePlatform` 같은 도메인 어노테이션으로 가드 의도가 컨트롤러 메서드 시그니처에 노출됨. 코드 리뷰가 쉬워짐.
- ✅ 도메인 어휘(플랫폼명) 가 그대로 코드 어휘에 노출되므로, 비즈니스 사람이 "A 플랫폼 구독자만" 이라고 말한 게 코드에서도 같은 단어로 보임.
- ⚠️ 모든 자원 서비스가 공통 라이브러리의 PlatformRole 에 의존. 스타터 enum 을 변경하면 모든 자원 서비스가 영향. 스타터 버저닝 + 호환성 정책이 더 중요해짐 — ADR-0001 과 연계.
- ⚠️ 등급(체험/프리미엄) 도입 시 enum 차원이 어색해질 가능성 있음. 그 시점에 별도 ADR 로 차원 분리 결정.
- ⚠️ 자원이 가드를 빼먹으면 무방비. Gateway 가 안 막아주니까 자원의 SecurityConfig 가 진짜 가드. PR 리뷰 체크리스트로 강제할 필요.
- 🔮 재검토 트리거:
  - 권한 차원이 3개 이상 (플랫폼 + 등급 + 부가서비스) 으로 늘어나면 enum 모델로 표현하기 어색해짐 → 권한 객체 / 정책 엔진(OPA) 도입 검토.
  - 회원 간 권한 위임 (조직 회원 vs 개인 회원, 관리자 위임) 이 필요해지면 정적 enum 으로는 부족 → 동적 권한 모델 ADR.
  - 외부 시스템과의 OAuth2 표준 통합이 강하게 요구되면 scope 모델로 회귀 검토 (B안).

## Lessons Learned
- 처음에 권한 어휘를 등급 (`ROLE_MEMBER` / `ROLE_USER`) 으로 시작했는데, 회사 라인업이 멀티 SaaS 로 늘어나면서 "어느 SaaS 의 회원인지" 를 표현할 방법이 없어서 곧 부족해졌음. 도메인 어휘(구독/플랫폼) 로 처음부터 시작했다면 좋았겠지만, 회사가 멀티 SaaS 가 될 거라는 걸 그 시점에 미리 알기는 어려웠음. 결국 어휘를 미리 정확히 맞추기보다 진화 가능한 구조(enum 으로 묶기) 를 두는 쪽이 현실적인 답이었음.
- ADR-0004 의 무력화 사고에서 가장 크게 얻은 건, 두 컴포넌트가 같은 어휘(권한 모델) 를 공유하는데 그걸 String 상수로만 관리하면 한쪽이 바뀐 걸 다른 쪽이 모른다는 점이었음. enum 으로 묶고 모든 컴포넌트가 같은 enum 을 import 하면 컴파일러가 동기화를 강제해주니까, 그게 곧 어휘 동기화 도구가 됨. 이번 ADR 의 핵심 결정 (스타터에 enum 단일 정의) 은 그 교훈을 구조로 박아넣은 것.
- 권한 어휘를 어디에 두느냐를 정하면 그걸 누가 박는지(IDP), 누가 검증하는지(자원 서비스), 누가 단지 전달만 하는지(Gateway) 의 역할이 따라서 정해짐. 어휘 정의 위치 결정이 책임 분배 결정의 출발점이 되는 셈. ADR-0004 와 ADR-0005 가 같이 그 그림을 완성했다고 보면 됨.
- 도메인 어휘를 코드 어휘로 그대로 가져오는 게 항상 최선은 아니지만, 도메인이 충분히 안정적이고 어휘가 자주 바뀌지 않는다면 가져오는 게 가장 깔끔함. 우리 회사가 세무/회계 도메인 SaaS 를 계속 만들 거라는 사실이 그 안정성을 보장해줘서 PlatformRole 어휘를 그대로 코드에 박을 수 있었음.

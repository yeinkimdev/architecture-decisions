# ADR-0001: 사내 공통 기능을 Spring Boot Starter 형태의 멀티 모듈 라이브러리로 제공

- Status: Accepted
- Date: 2025-06-02
- Deciders: 김예인
- Tags: #platform #standardization #boilerplate

## Context
- 신규 마이크로서비스를 도메인 단위로 분리하는 과정에서, 모든 서비스가 반복해서 작성해야 하는 코드(공통 응답 포맷, 비즈니스 예외 계층, 글로벌 Exception Handler, BaseEntity, ID 생성 전략, 암호화 유틸, JPA Auditing, Jasypt 설정값 암호화 등)가 빠르게 늘어나고 있었음.
- 같은 코드를 9개 서비스에 복사 → 일관성 깨짐 + 유지보수 분산 + 보안 정책 변경 시 여러 곳을 수정 해야하는 명확한 비용이 보였음.
- 새 서비스 부트스트랩 시간이 길어지는 것도 문제 (개발자가 본질이 아닌 설정/공통 코드에 시간을 씀).

## Decision
사내 공통 기능을 `core / data / web` 3개 모듈로 분리한 **Spring Boot Auto-Configuration 기반 사내 Starter 라이브러리**로 패키징하여 사내 사설 Maven 저장소에 배포한다. 각 서비스는 `api(...)` 의존성 한 줄만 추가하면 공통 기능이 자동 활성화된다.

## Alternatives Considered
- **A안 (선택됨): 사내 Starter 라이브러리**
  - 장점: 자동 구성으로 사용처 코드 0줄, 버전으로 정책 변경 일괄 전파, 일관된 응답/예외 포맷 강제 가능
  - 단점: 초기 설계 비용, 라이브러리 자체의 호환성/버저닝 책임 발생
- **B안: GitHub Template Repository로 공통 코드를 복사 시작**
  - 장점: 도입 비용 0, 각 팀이 자유롭게 변형 가능
  - 단점: 시간이 지나면 도메인별로 사본이 모두 갈라져서 "표준" 자체가 사라짐.
    이미 한 번 겪었던 실패 패턴.
- **C안: 공통 모듈을 단일 JAR로만 제공 (Auto-config 없음)**
  - 장점: 단순, 명시적
  - 단점: 사용처마다 `@Import`/`@ComponentScan`을 직접 써야 해서 실수 여지 + 부트스트랩 비용 그대로

## Consequences
- ✅ 체감상 새 서비스 부트스트랩이 1일 → 30분 수준으로 단축됨
- ✅ 응답·예외 포맷이 모든 서비스에서 일관 — 프론트/Backoffice 통합 비용 감소
- ✅ 보안 정책(BCrypt 라운드, RSA 키 사이즈 등) 변경 시 라이브러리 버전 업만으로 전파
- ⚠️ 라이브러리 자체의 SemVer를 엄격히 관리해야 함. Breaking change는 major 업 + 마이그레이션 가이드 필수 (사내 모든 서비스가 의존하는 라이브러리니까, 버전을 함부로 올리면 다 깨진다. 호환 안 되는 변경(Breaking change)이 들어가면 반드시 MAJOR 버전을 올리고(예: 1.x → 2.0), 사용자(타팀 개발자)가 어떻게 옮겨야 하는지 마이그레이션 가이드 문서를 같이 제공해야 한다.)
- ⚠️ **현재 한계**: Auto-Configuration이 동작하는데도 모듈별 README와 `@ConditionalOnProperty`를 아직 적용하지 않음. 신규 합류자가 빈 등록 흐름을 추적하기 어려운 비용을 치르고 있음
- 🔮 재검토 트리거: 모듈 수가 5개 이상으로 늘거나, 의존하는 서비스가 20개를 넘으면 BOM 패턴 도입 또는 모듈 단위 레포 분리 고려

## Lessons Learned
- **"공통화의 진짜 비용은 만들 때가 아니라 운영하면서 든다"** 는 점을 체감.
  라이브러리를 만든 직후에는 "공통 코드 추출"에만 집중했지만, 시간이 지나면서 (1) 신규 합류자 온보딩 비용, (2) "이 빈은 어디서 등록됐지?" 추적 비용, (3) 버전 업 시 사용 서비스 호환성 검증 비용이 점진적으로 발생.
- 다음 라이브러리를 처음부터 만든다면 **첫 릴리스 전에 반드시 함께 갖출 것**:
  - 모듈별 README (자동 등록되는 빈 목록 + 활성화 조건 + 끄는 방법)
  - 모든 자동 등록 빈에 `@ConditionalOnProperty` (`matchIfMissing = true`로 기본 ON, 명시적으로 OFF 가능)
  - `CHANGELOG.md`와 SemVer 정책 README

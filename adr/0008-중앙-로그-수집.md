# ADR-0008: 중앙 로그 수집 — Filebeat DaemonSet → Elasticsearch 직결

- Status: Accepted (dev 환경에서 운영 중 / staging·prod 미배포)
- Date: 2026-05-22
- Deciders: 김예인
- Tags: #observability #logging #elasticsearch #filebeat #kubernetes

## Context
- SSH 배포 시기부터 개발 환경의 로그를 보는 게 힘들었음. 서버에 직접 들어가서 로그 파일을 찾아 봐야 했고, 여러 인스턴스에 같은 서비스가 떠 있으면 어느 쪽을 봐야 하는지부터 막혔음. 그때부터 중앙 로그 수집기 만들어야 한다고 계속 요구했지만 우선순위에 밀려서 미뤄짐.
- K8s + ArgoCD 로 배포 방식이 바뀐 뒤에는 ArgoCD UI 에서 Pod 로그를 볼 수 있게 됐음. 처음에는 이걸로 충분해 보였는데 곧 한계가 드러남
  - Pod 가 새로 뜨면 이전 Pod 의 로그가 사라져서 장애 분석할 때 가장 보고 싶은 죽기 직전 로그가 자주 휘발됨.
  - Pod 마다 따로 들어가서 봐야 함. 한 요청이 여러 서비스를 거치면 흐름을 시간순으로 따라가기 어려움.
- 이 한계가 드러난 시점에 다시 중앙 로그 수집을 추진함. 이번에는 우선순위가 통과돼서 도입을 진행함.
- 이 ADR 은 ELK 채택과 Filebeat DaemonSet 구조의 결정 배경, 그 결정에 영향을 준 제약 (애플리케이션 로깅 라이브러리 호환성) 을 기록함.

## Journey

### Step 1 — SSH 배포 시기부터의 요구
- SSH 로 배포하던 시기부터 다른 개발자들이 로그 보기 힘들다는 게 가장 큰 부채였음.
- 중앙 로그 수집 도입을 여러 번 건의했지만 다른 작업 우선순위에 계속 밀림.

### Step 2 — K8s/ArgoCD 전환 후 ArgoCD UI 한계 인식
- K8s 도입과 ArgoCD UI 의 Pod 로그 뷰로 일부 해소되는 것처럼 보였음.
- 그러나 Pod 가 재기동되면 이전 Pod 의 로그는 클러스터에서 사라짐 (Pod 의 stdout/stderr 는 노드의 컨테이너 런타임 로그일 뿐, ArgoCD UI 가 영구 저장소가 아님).
- 분산 흐름 추적이 안 됨. Pod 마다 별도 탭/창에서 따로 봐야 함.
- 임시 해법이라 생각했던 ArgoCD UI 가 실제로는 기동중인 Pod 의 현재 로그만 보여주는 도구였고, 디버깅 시점이 한 박자 늦게 오는 운영 패턴과 안 맞았음.

### Step 3 — ELK 선정 (사내 ES 재활용)
- 거창한 비교 없이 ELK 로 결정함. 가장 익숙하고 가장 흔한 스택이라는 정도의 이유.
- 사내에 별도로 운영되는 Elasticsearch 인스턴스가 있었음 (다른 용도로 이미 설치되어 있던 자산). 클러스터 안에 ES 를 새로 띄울 필요 없이 기존 인스턴스에 인덱스만 추가하는 형태로 시작 가능했음. 이게 도입 비용을 크게 낮췄음.

### Step 4 — 인프라팀 권고 (Logstash) 가 있었지만 프로젝트 제약 (log4j2) 발견 → Filebeat 로 선회
- 인프라 담당자는 처음에 Logstash 사용을 권고함. 사내 ES 인스턴스 운영 관점에서 Logstash 가 자연스러운 그림이었음.
- 그 권고를 따라가려고 `logstash-logback-encoder` 를 logback 에 붙이는 형태를 검토함 (앱이 JSON + Logstash 호환 포맷으로 직접 출력 → Logstash 가 받아 ES 적재).
- 막상 프로젝트에 적용하려고 보니 로깅 라이브러리가 **log4j2** 였음. `logstash-logback-encoder` 는 logback 전용이라 그대로는 못 씀. log4j2 에서 같은 효과를 내려면 모든 서비스의 `log4j2.xml` 을 다 손봐야 했음.
- 결론: 애플리케이션 측은 그대로 두고 인프라 쪽에서 로그를 긁어가는 방식으로 선회함. 그래서 Filebeat 채택하게 됨.
- Filebeat 가 ES 로 직접 전송하는 형태로 단순화. 변환/enrichment 가 별로 필요 없었음 (Spring Boot 의 JSON 로그 + Filebeat 의 K8s 메타데이터 자동 부착으로 충분).
- Filebeat 의 장점이 자연스럽게 따라옴. 
  - 애플리케이션 코드/설정 변경 0. 컨테이너 stdout/stderr 만 정상적으로 나오면 K8s 가 알아서 노드의 `/var/log/containers/` 에 떨어뜨려 줌. Filebeat 는 그걸 읽기만 함.
  - 신규 서비스가 추가될 때마다 로그 수집 설정을 따로 하지 않아도 됨 (autodiscover).

### Step 5 — DaemonSet + autodiscover + namespace 필터 + K8s 메타 자동 부착
- 배치는 DaemonSet 으로, 노드당 Filebeat 1개가 그 노드의 모든 컨테이너 로그를 읽음.
- `autodiscover` + namespace 필터로 서비스 네임스페이스의 Pod 로그만 수집. 운영 시스템 (kube-system 등) 로그는 안 모음.
- `add_kubernetes_metadata` 프로세서로 K8s 라벨/네임스페이스/Pod명을 로그에 자동 부착. Kibana 에서 라벨 기반 필터링이 즉시 가능해짐.
- 인덱스 폭증 방지를 위해 헬스체크 (`/actuator/health*`) 메시지와 불필요 필드 (`agent.*`, `ecs.version` 등) 는 Filebeat 단계에서 drop.
- 인덱스 명: `{환경}-{yyyy.MM.dd}` (환경별 + 일자별 rolling).
- ArgoCD App-of-Apps 위에 Filebeat 도 매니페스트로 올림. 다른 서비스들과 동일한 GitOps 흐름으로 관리.

### Step 6 — 현 상태 (dev 운영, staging/prod 미배포)
- 현재 dev 환경에만 배포되어 운영 중. ArgoCD UI 한계가 해소됐고 Kibana 에서 통합 검색이 가능해짐.
- staging/prod 는 배포 전. 따라서 운영 부하 / 인덱스 용량 / 멀티라인 stacktrace 등 운영 튜닝 항목은 아직 손대지 않음.

## Decision
1. **ELK 스택을 사용한다.**
   - Elasticsearch / Kibana 는 사내에 별도 운영 중인 인스턴스를 재활용. 클러스터 안에 ES 를 띄우지 않음.
2. **로그 수집기는 Filebeat DaemonSet 으로 배치한다.**
   - 노드당 1개. 노드의 `/var/log/containers/` 를 hostPath 로 마운트해서 그 노드의 모든 컨테이너 로그를 수집.
   - sidecar 패턴은 사용하지 않음.
3. **Filebeat 가 Elasticsearch 로 직접 전송한다 (Logstash 생략).**
   - Spring Boot 의 JSON 로그 + Filebeat 의 `add_kubernetes_metadata` 로 변환/enrichment 요구 충족.
4. **수집 범위는 서비스 네임스페이스로 한정한다.**
   - `autodiscover` + namespace 조건 필터. 운영 시스템 로그는 수집 안 함.
5. **헬스체크 로그와 불필요 필드는 Filebeat 단계에서 drop 한다.**
   - `/actuator/health*` 메시지 drop, `agent.*` / `ecs.version` 등 메타 필드 drop.
6. **인덱스는 `{환경}-{yyyy.MM.dd}` 형식으로 환경 + 일자별로 분리한다.**
   - ILM / index template 은 비활성화. 운영 진입 시점에 정책 정의 예정.
7. **Filebeat 도 ArgoCD App-of-Apps 위에 올린다.**
   - 다른 서비스와 동일한 GitOps 흐름으로 관리.

## Alternatives Considered
- **A안: Logstash + `logstash-logback-encoder` (애플리케이션 직접 전송)**
  - 장점: 애플리케이션이 구조화 로그를 직접 ES 로. 변환 단계가 명시적이고 풍부함. 인프라팀 권고와도 일치.
  - 단점: 프로젝트의 로깅 라이브러리가 log4j2 라 직접 적용 불가. log4j2 에서 같은 효과를 내려면 모든 서비스의 `log4j2.xml` 변경 필요. 서비스 수가 늘어나는 시점에 그 비용이 컸음.
  - 결론: 권고를 따라 시도했다가 프로젝트 제약 확인 후 제외.
- **B안 (선택됨): Filebeat DaemonSet → Elasticsearch 직결**
  - 장점: 애플리케이션 코드/설정 변경 불필요. K8s 가 stdout/stderr 를 자동으로 노드 파일로 떨어뜨려 주고 Filebeat 가 그걸 읽음. autodiscover 로 신규 서비스 자동 픽업.
  - 단점: ES 인덱스 mapping 충돌 가능성 (필드 타입 자동 추론 위험). 변환/enrichment 가 Filebeat 의 프로세서 수준으로 제한됨 (복잡한 변환은 향후 Logstash 또는 ES ingest pipeline 필요).
- **C안: Loki + Promtail**
  - 장점: 라벨 기반 인덱싱으로 인덱스 비용이 낮음. Grafana 와의 통합이 자연스러움.
  - 단점: 사내에 ES 가 이미 운영 중이어서 Loki 를 새로 들여올 만큼의 동기는 부족했음.
  - 결론: 제외.

## Consequences
- ✅ Pod 가 종료되어도 로그가 ES 인덱스에 보존됨. 장애 분석에서 가장 필요한 "죽기 직전 로그" 가 살아있음.
- ✅ Kibana 에서 여러 Pod / 서비스 / 네임스페이스의 로그를 시간순으로 통합 검색 가능. 분산 흐름 추적이 가능해짐.
- ✅ 애플리케이션 측 코드/설정 변경 0. log4j2 그대로 유지하면서 중앙 수집 달성.
- ✅ 신규 서비스 추가 시 로그 수집 설정 작업 0. autodiscover + namespace 필터가 자동 픽업.
- ✅ K8s 라벨/메타데이터가 로그에 자동 부착되어 라벨 기반 필터링 가능.
- ✅ Filebeat 도 ArgoCD App-of-Apps 위에서 다른 서비스와 동일한 흐름으로 배포/관리됨.
- ⚠️ ES 인스턴스에 외부 의존. 가용성 / 용량 / 인덱스 정책은 인프라 측 책임. ES 다운 시 로그 수집 중단 (Filebeat 의 디스크 큐로 일부 버퍼링 가능).
- ⚠️ ILM 미설정. 인덱스가 무한히 쌓이는 구조. 운영 진입 전 retention/롤오버 정책 정의 필수.
- ⚠️ Filebeat → ES 직결이라 변환/enrichment 가 Filebeat 프로세서 수준으로 제한됨. 복잡한 파싱 (멀티라인 stacktrace 등) 이 필요해지면 Filebeat `multiline.pattern` 추가 또는 Logstash 도입 검토 필요.
- ⚠️ 운영 환경 (staging/prod) 미배포 상태. 부하 / 디스크 폭증 / 멀티라인 / mapping 충돌 등 운영 튜닝 항목이 아직 검증 안 됨.
- 🔮 재검토 트리거:
  - ES 디스크 폭증 → ILM (hot-warm-cold) 정책 정의.
  - 멀티라인 Java stacktrace 가 깨져서 들어옴 → Filebeat `multiline.pattern` 추가 또는 Logstash 도입.
  - 검색 성능 저하 / mapping 충돌 → index template 정의, shard 튜닝.
  - 비용/성능 한계 도달 → Loki / Tempo 같은 가벼운 대안 재검토.
  - 보안 요구 강화 (감사 로그 분리, 접근 통제) → 별도 인덱스 / 별도 ES 클러스터.

## Lessons Learned
- 중앙 로그 수집이 필요하다고 SSH 시기부터 계속 의견을 냈었지만 우선순위에 밀려서 못 했었다. K8s + ArgoCD 전환 후 ArgoCD UI 의 로그 뷰가 임시 해법처럼 잠깐 통하는 듯 보였는데, Pod 가 죽으면 로그도 같이 사라진다는 본질적 한계가 곧 드러났다. 임시 해법이 충분해 보일 때도 어떤 한계가 곧 드러날지 한 번은 짚고 넘어가야 한다는 걸 배웠다.

- ELK를 선택한 건 특별한 이유가 있었던 게 아니라 단순히 가장 널리 쓰이는 스택이었기 때문이고, 게다가 회사에서 이미 Elasticsearch 인스턴스를 운영하고 있었다는 점이 결정적이었다. 도구 자체보다 지금 가진 자산을 활용하는 게 의사결정 과정에서 더 효율적일 수 있다는 걸 느꼈다.

- 처음에 인프라 담당자가  권고한 방식은 Logstash 였고, 그 권고를 그대로 따라가려고 `logstash-logback-encoder` 를 검토했다. 그런데 실제로 적용해 보려고 하니 프로젝트가 log4j2 라 그대로 들어맞지 않았다. 권고가 잘못된 게 아니라 인프라 쪽에서 보는 그림과 우리 프로젝트 사정이 달랐던 거고, 그걸 적용 단계에서 빨리 알아챈 게 다행이었다. 권고를 받으면 일단 우리 프로젝트에 한 번 대입해 보는 단계는 빠뜨리면 안 되겠다고 느꼈다.

- 운영 튜닝은 아직 못 했다. 운영 환경에 배포되지 않았기 때문에 부하 / 디스크 폭증 / 멀티라인 stacktrace 같은 항목이 모두 미검증이다. 일단 dev 에서는 충분히 동작하고 있고, 운영 진입 시점에 드러날 이슈들은 그 시점에 별도 ADR 로 추가 작성할 예정이다.

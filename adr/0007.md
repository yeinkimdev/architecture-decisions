# ADR-0007: GitOps 도입 — Jenkins(Build) + ArgoCD(Deploy)

- Status: Accepted
- Date: 2026-05-20
- Deciders: 김예인
- Tags: #gitops #cicd #kubernetes #argocd #jenkins #kustomize

## Context
- 사내 SaaS 라인업의 신규 서비스들을 배포할 자동화가 필요했음. 처음에는 Jenkins + Docker 조합으로 JAR 를 빌드해서 SSH 로 원격 서버에 전송하고, 거기서 스크립트로 다시 실행시키는 형태였음. 분명히 자동화는 됐는데, 일을 하다 보면서 이게 정말 CI/CD 인가라는 의문이 계속 들었음.
- 셀프호스팅 Git 을 쓰고 있어서 GitHub Actions 같은 SaaS CI 는 처음부터 후보가 아니었고, 그 시점에 가장 익숙한 도구가 Jenkins 였음. 그래서 Jenkins 를 그대로 들고 왔지만 SSH 배포의 한계 (롤백 부재, 환경 간 일관성 부족, 배포 상태 가시성 부족) 가 누적되면서 다음 단계가 필요해짐.
- 인프라 담당자가 새로 합류하면서 인프라 영역의 결정 비용이 낮아졌고, 이참에 K8s 도입을 직접 건의해서 진행하기로 함. 러닝커브를 알면서도 한 번은 정확한 의미의 CI/CD 를 구축해 보고 싶다는 동기가 컸음.
- 이 ADR 은 그 전환의 배경과 GitOps 구조 (Jenkins(Build) / ArgoCD(Deploy) / manifests repo 분리 / Build Once Deploy Many) 에 대한 결정을 기록함.

## Journey

### Step 1 — 첫 자동화: Jenkins + Docker + SSH
- Jenkins 가 Gradle 로 JAR 를 빌드 → SSH 로 원격 서버에 전송 → 원격 스크립트로 재실행하는 형태.
- 한계가 누적됨:
  - 배포 실패 시 롤백 절차가 없음. 원격 서버에 들어가서 이전 JAR 로 수동 복귀해야 함.
  - 환경 간 일관성 보장 못 함. 각 서버의 OS/JDK/네트워크 상태에 의존.
  - 배포 상태 가시성 부족. SSH 명령 종료가 정상 동작을 의미하지 않음.
  - 중앙 로그 수집 힘듦
- 자동화는 됐는데 CI/CD 라고 부르기에는 부족하다는 생각이 맴돌았음.

### Step 2 — 인프라 담당자 합류 → K8s 도입 건의
- 이전에는 인프라 결정의 비용이 너무 컸음 (인프라 영역을 함께 고민할 사람이 없었음). 담당자 합류로 그 비용이 낮아짐.
- "이참에 K8s 가보자" 라고 직접 건의. 처음 해보는 영역이지만 학습 의지로 가기로 함.
- K8s 결정과 거의 동시에 배포 도구도 ArgoCD 로 그림이 그려졌음. 깊은 비교 없이 "K8s 갈 거면 ArgoCD" 라는 직관에 가까웠음.

### Step 3 — Jenkins(Build) / ArgoCD(Deploy) 분리, manifests repo 처음부터 분리
- Jenkins 는 빌드 + 이미지 push 까지만 담당. 클러스터 권한을 들고 있지 않음.
- ArgoCD 가 manifests repo 를 watch 해서 클러스터 상태를 동기화 (pull 방식).
- 앱 코드 repo 와 manifests repo 를 처음부터 분리. 정확한 근거가 있는 건 아니고, 이건 별도로 관리되어야 할 것 같다는 직관이었음.
- 사후 정당화: 두 repo 의 변경 주기와 권한 도메인이 다름. 앱 코드는 빌드/테스트 → 이미지로 끝나는 사이클, manifests 는 클러스터 상태 자체에 대한 선언.

### Step 4 — "환경별 빌드는 이상하다" 깨달음 → Build Once, Deploy Many
- 초기 계획은 dev 환경부터 만들고 staging/prod 는 차차 진행하기로 함. 한 번에 다 끝낼 일이 아니라고 봤음.
- dev 가 자리잡고 staging/prod 흐름을 그리다가 어색한 점이 보임:
  - 같은 소스를 환경마다 다시 빌드하면 동일한 결과물이라는 보장이 없음.
  - prod 에서 발견된 버그를 dev 에서 재현할 때 내가 본 그 이미지라는 확신이 깨짐.
- 결론을 바꿈:
  - dev 잡에서만 빌드하고 이미지를 푸시.
  - staging/prod 잡은 빌드 없이 dev overlay 의 태그를 promote 스크립트로 복사.
- 패턴 이름 (Build Once, Deploy Many) 은 나중에 알았고, 이 시점에는 그냥 이렇게 해야한다는 직관적 결론이었음.

### Step 5 — App-of-Apps 처음부터
- 서비스 수가 늘어날 거라는 건 처음부터 알고 있었음. ArgoCD Application 을 서비스 × 환경마다 일일이 등록하면 그 자체가 운영 부담이 됨.
- 환경별 루트 Application 1개씩 (dev / staging / prod) 두고, 루트가 `argocd/applications/{env}/` 폴더를 watch 하는 형태로 처음부터 구성.
- 신규 서비스 추가는 폴더에 yaml 한 장 추가하면 끝나는 비용으로 확정됨.

## Decision
1. **빌드는 Jenkins, 배포는 ArgoCD 로 분리한다.**
   - Jenkins 는 빌드 + 이미지 push 까지만. 클러스터에 직접 손대지 않는다.
   - ArgoCD 가 manifests repo main 브랜치를 watch 해서 클러스터 상태를 동기화 (pull 방식).
2. **앱 코드 repo 와 manifests repo 를 분리한다.**
   - Jenkins 가 이미지 push 후 manifests repo 의 해당 overlay 의 image tag 를 자동 PR/머지로 갱신한다.
3. **이미지 태그는 `{git-sha7}-b{buildNumber}-{timestamp}` 로 고정한다.**
   - 불변 + 추적성. `latest` 는 사용하지 않는다.
4. **환경별 잡을 분리하고, staging/prod 는 빌드 없이 이미지 태그를 promote 한다.**
   - dev 잡: 빌드 + push + dev overlay 갱신.
   - staging/prod 잡: 빌드 없음. `promote-image.sh` 로 이전 환경 overlay 의 태그를 복사.
5. **ArgoCD 는 App-of-Apps 패턴으로 구성한다.**
   - 환경별 루트 Application 이 `argocd/applications/{env}/` 폴더를 watch.
6. **ArgoCD 동기화 정책은 자동 + 자가복구 + prune.**
   - `automated.selfHeal=true`, `automated.prune=true`.
   - HPA 가 관리하는 `spec.replicas` 는 `ignoreDifferences` 로 드리프트 예외.
7. **매니페스트 추상화는 Kustomize 로 한다.**
   - base + overlays/{env}. Helm 은 채택하지 않음 (Alternatives D안 참조).

### 전체 흐름
```
[앱 repo: dev 브랜치 push]
        │
        ▼
[Jenkins CI]
  - Gradle build + test
  - Jib 으로 컨테이너 이미지 빌드 & 레지스트리 push
  - 이미지 태그: {git-sha7}-b{buildNum}-{timestamp}
        │
        ▼
[Jenkins: 매니페스트 repo 자동 PR]
  - apps/{svc}/overlays/dev/kustomization.yaml 의 image tag 갱신
  - feat 브랜치 자동 생성 → PR 자동 생성 → 자동 머지
        │
        ▼
[ArgoCD: pull-based sync]
  - 매니페스트 repo main 브랜치 watch
  - selfHeal=true (수동 변경 자동 복구)
  - prune=true (Git 에 없는 리소스 정리)
        │
        ▼
[dev 클러스터 배포]
[dev → main 머지 → staging promote 잡]
  - 빌드 없음, dev overlay 의 이미지 태그를 staging overlay 로 복사
  - 동일하게 ArgoCD 가 staging 클러스터로 sync
[staging 검증 → prod promote 잡]
  - 동일한 promote 로직, prod overlay 갱신
```

## Alternatives Considered
- **A안: GitHub Actions / GitLab CI 도입**
  - 장점: SaaS CI 의 운영 부담 없음. 워크플로우 yaml 만으로 정의.
  - 단점: 셀프호스팅 Git 사용 중. Git 호스팅을 마이그레이션해야 하는 비용이 큼.
  - 결론: 제외.
- **B안 (선택됨): Jenkins(Build) + ArgoCD(Deploy) + manifests repo 분리**
  - 장점: 빌드/배포 책임이 명확하게 나뉨. ArgoCD selfHeal 로 운영 안정성 확보. 앱 변경과 인프라 변경의 권한/리뷰가 자연스럽게 분리됨.
  - 단점: 도구 두 개의 학습/운영 부담. credential 분산. 자동 PR 로 인한 manifests repo PR history 증가.
- **C안: Push 방식 (Jenkins 가 kubectl apply)**
  - 장점: 단일 도구. 학습 비용 적음.
  - 단점: Jenkins 가 클러스터 권한을 직접 들고 있어야 함. 클러스터 상태와 Git 의 동기화가 보장되지 않음. 수동 변경이 흔적 없이 사라지지 않음. ArgoCD selfHeal 이 주는 운영 안정성을 포기.
  - 결론: 잠깐 고려 후 제외.
- **D안: Helm + ArgoCD**
  - 장점: 차트 생태계가 풍부함. 외부 컴포넌트는 차트 그대로 사용 가능.
  - 단점: 템플릿 엔진 학습이 추가됨. values 파일 분기가 늘어나면 복잡도가 빠르게 증가. K8s 자체 학습 곡선에 도구 곡선까지 쌓고 싶지 않았음.
  - 결론: Kustomize 채택.
- **E안: Mono-repo (앱 + manifests 한 곳)**
  - 장점: 변경 추적 단순. PR 하나로 코드 + 매니페스트 동시 변경 가능.
  - 단점: 권한 도메인이 섞임. 빌드 트리거와 배포 트리거가 한 repo 에 얽힘. 권한 분리 필요해지면 회수 비용이 큼.
  - 결론: 처음부터 분리.

## Consequences
- ✅ 같은 이미지가 dev → staging → prod 로 그대로 흐름. 환경 일관성 보장됨.
- ✅ ArgoCD selfHeal 로 클러스터의 수동 변경이 자동 복구됨. 운영 안정성 확보.
- ✅ 이미지 태그에 git sha + buildNumber + timestamp 가 박혀 있어 어떤 빌드인지 즉시 식별 가능.
- ✅ 신규 서비스 추가 비용이 yaml 한 묶음으로 줄어듦 (App-of-Apps + Kustomize 템플릿).
- ✅ "정확한 의미의 CI/CD" 라는 처음 동기를 만족시킴.
- ⚠️ ArgoCD / K8s 자체의 학습 · 운영 비용이 작지 않음. 인프라 담당자 합류가 없었으면 시작이 어려웠을 결정.
- ⚠️ 자동 PR/머지가 매 빌드마다 생성되어 manifests repo 의 PR/브랜치 history 가 빠르게 길어짐. 정리 정책이 별도로 필요.
- ⚠️ "클러스터 수동 변경 절대 금지" 라는 팀 합의가 selfHeal 의 전제. 임시 변경도 자동으로 사라지므로 운영 문화의 변화가 필요했음.
- ⚠️ credential 관리 복잡도 증가 (Jenkins credentials + 서비스별 secret + 레지스트리 토큰).
- 🔮 재검토 트리거:
  - manifests repo 의 PR history 가 운영 부담이 되는 시점 → 자동 머지 squash / 정리 정책 ADR.
  - Kustomize 만으로 표현이 어려운 외부 컴포넌트가 늘어나면 Helm 병용 검토.
  - 멀티 클러스터 / 멀티 리전이 요구되면 ArgoCD ApplicationSet / Hub-Spoke 구조 ADR.

## Lessons Learned
- 처음 자동화를 만들었을 때는 "어쨌든 자동화가 됐으니 됐다" 고 생각했는데, 시간이 지날수록 SSH 배포의 한계가 점점 크게 느껴졌다. 롤백을 못 한다는 게 가장 컸고, 환경마다 다른 결과가 나올 수 있다는 점도 운영 입장에서 부담이었다. 다음 단계로 가는 데 가장 큰 추진력이 된 건 거창한 이론보다 "이건 좀 아닌 것 같다" 는 막연한 회의감이었던 것 같다.

- 도구 선정에서 Jenkins 를 고른 건 사실 "그 시점에 내가 가장 잘 알아서" 였다. 솔직히 그 정도 이유였고, 거창한 비교 분석을 하지 않았다. 그래도 결과적으로 큰 후회가 없는 건 — 어떤 도구를 골랐든 본질적으로 비슷한 그림이 됐을 거라고 지금은 생각하기 때문이다. CI 도구는 어느 선까지는 대체 가능한 영역이고, 진짜 결정은 "Jenkins 냐 GitHub Actions 냐" 가 아니라 "ArgoCD 를 쓸 거냐 안 쓸 거냐" 쪽에 있었던 것 같다.

- repo 를 처음부터 분리한 건 직관이었는데, 시간이 지나고 보니 권한과 리뷰 흐름이 자연스럽게 분리되어서 결과적으로 잘한 결정이었다. 다만 직관이 항상 맞는 건 아니라는 점도 알고 있다. 이번에는 운이 좋았던 케이스에 가깝고, 다른 결정에서도 "직관 → 사후 정당화" 패턴이 통할 거라고 일반화하지는 않으려고 한다.

- "환경별로 빌드를 다시 하면 같은 소스가 같은 결과물이라는 보장이 없다" 는 건, staging/prod 자동화를 그림으로 그리다가 우연히 알아챈 거였다. 처음부터 명확하게 보였던 게 아니라 손으로 그리다 보니까 어색한 부분이 드러난 거였고, 그래서 처음부터 완벽하게 설계하려고 하기보다 일단 그려보고 어색한 데를 발견하는 식의 접근이 나한테는 잘 맞는다는 걸 느꼈다.

- App-of-Apps 를 처음부터 채택한 건 후속 결정에 큰 영향을 줬다. 서비스가 늘어날 때마다 매니페스트 등록 비용이 거의 없었고, Filebeat 같은 인프라 컴포넌트도 같은 패턴 위에 올라갈 수 있었다. 처음에 한 단계 추상화를 넣어둔 비용이 나중에 여러 번 회수된 셈이다.

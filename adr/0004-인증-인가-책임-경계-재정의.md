# ADR-0004: 인증/인가 책임 경계 재정의 — Gateway는 인증, Service는 인가

- Status: Proposed
- Date: 2026-05-19
- Deciders: 김예인
- Tags: #authorization #microservices #gateway #idp #journey

## Context
- ADR-0003 Step 8 에서 IdP 를 분리하면서 "인증 책임은 IdP, 인가 책임은 Gateway" 로 정리했었음. 그런데 백오피스 신규 작업을 준비하다가 그 Gateway 의 인가 필터가 사실상 무력화되어 있다는 사실을 우연히 발견함.
  - 권한 필터가 한 가지 회원 식별 헤더(레거시 컨벤션) 를 기준으로 인증 여부를 판단하는데, 인증 필터 쪽은 시간이 흐르면서 식별자 헤더를 신규 컨벤션으로 옮겨놓은 상태였음. 두 필터가 같은 어휘를 공유해야 하는데 한쪽만 갱신되면서, 인가 필터는 매번 "헤더 없음 → 검증 스킵" 으로 빠져나가고 있었음. 코드는 살아있는데 정책은 죽어있는 상태.
- 그 사실에 더해, 인가 필터를 정상 동작시키면 **모든 일반 회원이 보호 경로에서 차단되는** 더 큰 정합성 문제도 같이 드러났음. IdP 가 일반 회원에게 발급하는 클레임(구독 서비스 코드 — `ROLE_PLATFORM_X` 등) 과 Gateway 가 검증하는 권한 어휘(`ROLE_MEMBER` / `ROLE_ADMIN`) 가 어긋나 있어서, 권한 필터가 살아나는 순간 모든 회원이 403 으로 막힘. **무력화가 우연히 정합성 깨짐을 가리고 있었음.**
- 이걸 그대로 두면, 권한이 더 세분화될 미래(체험/베이직/프리미엄/부가서비스, 권한 위임 등) 에 Gateway 정책 폭발 + 회귀 위험이 커짐.
- 자원 컨텍스트 의존 인가(소유권 검사, 본인 글만 수정, 시간 제한 접근 등) 는 Gateway 가 본질적으로 못 하는 영역. path 매칭만으로는 "이 글의 작성자가 맞는가" 를 판단할 수 없음.
- 즉, ADR-0003 Step 8 이 인증을 IdP 로 빼낸 결정이었다면, 이번 ADR 은 그 후속 — **이번에는 인가 책임을 Gateway 에 계속 두지 않고, 실제 자원을 가진 서비스 쪽으로 옮기기로 함.**

## Journey (Gateway 인증·인가의 발자취)
> ADR 정통 형식은 아니지만, ADR-0003 의 Decision Timeline 패턴과 같은 의도로 시간축에 결정 발자취를 남김.
> 지금의 무력화 사고가 단발성 버그가 아니라 누적된 결정의 결과라는 점을 후행으로도 추적할 수 있도록.

### Step 1 — Gateway 신설 (SCG 채택, 인증 몰빵 의도)
- MSA 도입 자체가 처음이라 게이트웨이 선정 기준이 명확하지 않았음. 가장 범용적이고 안전하다고 판단한 Spring Cloud Gateway 채택. 다른 후보들(Zuul/Kong/AWS API Gateway/직접 구현)은 깊이 검토하지 않음.
- **인증은 무조건 Gateway 에 몰빵한다** 는 게 처음의 직관이었음. "각 서비스가 토큰 검증 로직을 따로 들고 있을 이유가 없다" 는 단순한 출발.

### Step 2 — 첫 인증 필터 (단일 필터)
- 처음엔 인증과 인가를 한 필터에 묶어둠. 토큰 파싱·만료 체크·권한 부여까지 한 번에 가능하도록 설계.
- 이 시점엔 권한이 단순했기 때문에(로그인 여부 정도) 한 필터로도 충분했음.

### Step 3 — 인증/인가 필터 분리 (JWT claim 설계가 트리거)
- JWT claim 을 본격적으로 설계하면서 **인증(이 토큰이 진짜인가) 과 인가(이 토큰이 이 자원에 접근 가능한가) 가 다른 책임** 이라는 걸 깨달음. 한 필터에 묶여있으니 권한 정책이 늘 때마다 인증 코드까지 함께 손대야 했음.
- 두 필터로 쪼갬: `AuthenticationFilter` (HIGHEST_PRECEDENCE) → `AuthorizationFilter` (HIGHEST_PRECEDENCE + 200).
- 이 과정에서 GlobalFilter / GatewayFilter 의 적용 순서, 도메인별 URI prefix 매칭이 의도대로 동작하지 않아서 시행착오가 많았음. 글로벌 필터를 따로 만들었다가, route 단위 필터로 옮겼다가, 다시 합치는 등 여러 번 갈렸음. (정확한 변천은 지금 회상으로는 흐릿함 — 코드를 다시 따라가야 정확. ADR 작성 시점의 한계로 인정하고 적어둠.)

### Step 4 — 헤더 모델 결정: user / member / role (지금과 정반대 모델)
- 이때 정한 명명 규칙이 지금 헤더 혼란의 근본 원인 중 하나라서 자세히 남김.
- **외부 사용자 = `user`, 내부 사용자(백오피스 운영자) = `member`** 로 명명. 왜냐하면:
  - 운영자도 audit (`createdBy` / `modifiedBy`) 가 필요. 운영자가 LDAP 으로 인증되어도 그 식별자가 회원 DB 에 한 행 있어야 audit 외래키가 깔끔하게 박힘. (팀 내 개발자들이 원했음.)
  - 그래서 운영자도 membership DB 에 기록하되, role 로 외부/내부를 구분하려는 모델.
  - role 테이블도 별도로 운영. 권한이 코드와 DB 양쪽에 있는 구조.
- 헤더는 그 모델을 그대로 반영 — 외부 사용자(회원) / 내부 사용자(운영자, 또는 통합 식별) / 식별 번호 별도 컬럼 의 세 가지 컨벤션으로 분기.
- **이 모델은 이후 다른 결정들로 인해 의미가 바뀌었지만 (Step 5 에서 IdP 분리 / Step 6 에서 운영자 LDAP 단일화 결정), 헤더 이름은 코드 곳곳에 남아있었음.** 외부 사용자 헤더가 "회원의 로그인 아이디" / 내부 사용자 헤더가 "회원의 고유 식별키" 로 의미가 바뀐 채 그대로 사용. role 테이블도 사라짐.
- 즉 헤더 어휘가 데이터 모델보다 늦게 따라온 잔재. 이번 ADR 의 무력화 사고도 이 잔재 위에서 발생.

### Step 5 — IdP 분리 (제품 의사결정 변경, 담당자 변경)
- 사내 SaaS 라인업이 늘어나면서 **단일 로그인(SSO)** 요구가 들어옴. 제품 의사결정 주체가 바뀌면서 OAuth 2.0 기반 IdP 분리로 가닥이 잡힘.
- 인증 서버(IdP) 를 별도 서비스로 신설. 로그인 처리/토큰 발급은 다른 담당자가 인수. 본인 롤은 GitOps / 인프라 쪽으로 이동.
- **여기서 책임 단절이 발생** — IdP 가 RS256 으로 발급하기 시작하면서 Gateway 의 옛 HS256 검증기 클래스와 그 부산물 DTO 가 데드 코드가 됨. 그러나 즉시 정리되지 않음. 본인이 만든 코드라 다른 담당자가 손대지 않았고, 본인은 다른 롤이라 적극적으로 제거 PR 을 제출하지 않음.
- 결과적으로 게이트웨이에 두 시대(HS256 + RS256) 의 검증 코드가 공존. 운영에는 RS256 만 쓰이지만 옛 코드가 같은 패키지에 살아있는 상태.
- ADR-0003 Step 8 에 "인증 책임은 IdP, 인가 책임은 Gateway" 로 적었지만, 그 인가 책임이 어떻게 운영되어야 하는지는 그 시점에 명확히 합의되지 않음.

### Step 6 — 서비스 분류 헤더 + 거친 인가 도입 (통합 관리자 요구)
- 관리자는 LDAP 단독 인증 + 통합 관리자는 사내 모든 플랫폼을 한 곳에서 관리 라는 두 요구가 들어옴. 즉 운영자는 IdP 와 분리하되, 게이트웨이가 플랫폼별 라우팅은 알아야 했음.
- Gateway route 마다 서비스 분류 헤더 (`x-service-type` 류) 를 자동 부착. 이를 받아서 `AuthorizationFilter` 가 거친 인가(서비스 단위)를 수행:
  - 운영자 전용 서비스 분류 → 관리자 권한만 허용
  - 회원 접근 가능 서비스 분류 → 회원 / 관리자 권한 허용
- 이 정책이 만들어진 이유는 **관리자가 모든 플랫폼을 한 게이트웨이로 들어와야 한다** 는 통합 요구 때문. 그러나 이 정책이 사실상 어떤 토큰을 가정하는지(IdP 가 발급하는 토큰의 권한 어휘) 는 명시되지 않음 — 이게 Step 8 에서 깨지는 정합성의 씨앗이 됨.

### Step 7 — 데드 코드의 누적 (책임의 모호함)
- Step 5 의 옛 JWT 검증기 / DTO 외에도, 그 검증기를 노출하는 디버깅 컨트롤러와 그 엔드포인트들이 살아있는 상태로 남음. 운영 환경 노출 위험까지 누적.
- 더 이상 어떤 컴포넌트에서도 채우지 않는 레거시 식별 헤더 상수도 `AuthorizationFilter` 만 읽고 있어서, 사실상 항상 null 인 헤더로 인가 결정을 시도. (Step 8 에서 폭발.)
- 코드 베이스 어디에도 "이건 deprecated 이므로 언제까지 정리하겠다"는 표식이 남아있지 않음. **데드 코드 관리 책임이 어느 담당자에게도 명시적으로 부여되지 않은 게 근본 원인.**

### Step 8 — 무력화 사고 발견 (이번 ADR 의 트리거)
- 백오피스 신규 작업 (게시판 관리 / 도메인 자료 관리 / RBAC) 을 준비하다가, 게이트웨이 코드 베이스를 다시 들여다보면서 발견. 우연히. 의도적 보안 점검도 아니고 운영 사고도 아님.
- `AuthorizationFilter` 가 참조하는 레거시 식별 헤더가 항상 null 이라 매번 "인증 정보 없음 → 다음 필터로" 로 빠져나가고 있음을 코드 리딩으로 확인. 즉 **권한 검증 자체가 매 요청 스킵**.
- 더 충격적인 건 그 필터를 살리려고 헤더만 정정하면 IdP 가 발급하는 권한 어휘와 안 맞아서 모든 회원이 차단된다는 점. (데드코드가 운영을 살리고 있던 셈.)

### Step 9 — 결정 (이번 ADR)
- 이번 사고를 "헤더 키 정정" 으로만 끝내면, 같은 류의 사고가 형태만 바꿔 다시 발생할 가능성이 높음. **두 컴포넌트(인증/인가) 가 같은 어휘(권한 모델) 를 공유하는데 그게 컴파일러로 강제되지 않으면 미래에 반드시 또 어긋남.** Step 4 에서부터 이어진 헤더 어휘 잔재가 그 증거.
- 따라서 이번에는 헤더 키 정정 수준의 패치가 아니라, **권한 어휘를 자원 옆으로 옮겨서 자원이 자기 인가를 직접 정의** 하는 구조로 전환. Gateway 는 권한 어휘를 모르고, 자원 서비스가 단일 진실(IdP 클레임)을 보고 자기 정책을 적용.
- 권한 이름과 의미는 IdP 클레임 기준으로 ADR-0005 (구독 단위 권한 모델) 로 분리. 운영자 인증 경계의 의도적 분리는 ADR-0006 (Internal Admin Console) 로 분리.

## Decision
1. **Gateway 는 인증만 책임진다.** IdP 발급 토큰 검증 + 검증된 클레임을 표준 헤더 (회원 식별자 + 권한 어휘) 로 다운스트림에 전달하는 것까지가 Gateway 의 책임 경계. 단일 진실은 IdP 클레임, 표준 헤더는 그 사본에 가까움.
2. **인가는 자원 서비스 각자가 책임진다.** Spring Security `SecurityFilterChain` + `@PreAuthorize` 한 줄로 자원 옆에 인가 가드.
3. Gateway 의 거친 인가 정책(`AuthorizationFilter` 의 admin-only / member-accessible 분류) 은 폐기. 권한 모델 어휘를 Gateway 에 새기지 않는다.
4. 자원 서비스는 헤더의 권한 클레임만 신뢰한다(Gateway 가 검증한 토큰의 결과). 단일 진실은 IdP 가 박아준 클레임.
5. 권한 어휘 통일성은 사내 공통 스타터 라이브러리(ADR-0001) 의 enum 한 곳에서 단일 정의 — ADR-0005 로 위임.
6. Step 5/7 의 데드 코드(옛 JWT 검증기·DTO, 디버깅 컨트롤러와 엔드포인트, 사용되지 않는 레거시 식별 헤더 상수) 는 이번 정리 PR 에서 함께 제거. 데드 코드를 의도적으로 남기는 경우 코드 베이스 어디에든 "EOL 일정 + 책임자" 가 명시돼야 함을 컨벤션화.

## Alternatives Considered
- **A안: Edge Authorization (Gateway 가 모든 인가)**
  - 장점: 자원 서비스가 인가 코드 0줄. 정책이 한 곳에 모임.
  - 단점: 권한이 추가될 때마다 Gateway 코드 변경/재배포. path 매칭 룰 폭발. 자원 컨텍스트 의존 인가 불가능. **정책이 무력화돼도 자원 쪽에서 알아챌 신호가 없음** — 이번 사고가 그 증거.
- **B안: IdP 가 일반 회원에게 추상 등급 권한(예: `ROLE_MEMBER`) 도 자동 부여**
  - 장점: 변경 한 줄 (IdP 의 권한 매핑에 추가 부여). Gateway 정책 그대로 유지.
  - 단점: 권한 모델이 두 층(추상 등급 + 구독 권한) 으로 쪼개짐. 어휘 흐려짐 — "그래서 ROLE_MEMBER 가 뭔데?" 가 매번 발생. 미래에 등급(베이직/프리미엄) 이 추가되면 어디에 박을지 또 정해야 함. **Step 4 에서 이미 한 번 흐려진 어휘를 다시 만드는 셈.**
- **C안 (선택됨): Decentralized Authorization (인가를 자원 서비스로)**
  - 장점: 권한 진화 시 Gateway 무변경. 자원 컨텍스트 의존 인가 가능. 자원 자율성 ↑. 정책 위반이 즉시 401/403 으로 가시화 — 죽은 정책이 오래 살아남기 어려움.
  - 단점: 모든 서비스가 자기 인가 가드를 직접 짜야 함. 일관성 보장이 컨벤션에 의존 → 공통 어노테이션/헬퍼(ADR-0005 의 `@RequirePlatform` 류) 도입으로 보완.

## Consequences
- ✅ 권한 추가/변경 시 Gateway 무변경. 자원 옆 한 줄로 끝.
- ✅ 자원 컨텍스트 인가(소유권, 본인 데이터, 시간 제한) 가 가능해짐. 게이트웨이가 못 하는 영역.
- ✅ "정책이 죽어있는데 살아있는 척" 위험 제거. 자원 서비스에서 권한 검증 실패는 곧장 401/403 으로 떨어지므로 즉시 알 수 있음.
- ✅ Gateway 가 단순해지면서 책임 경계가 명확 — 토큰 검증 + 헤더 변환만. 새 인증 흐름이 추가돼도 Gateway 의 책임은 그대로.
- ⚠️ 자원 서비스가 늘어날 때마다 인가 컨벤션을 따라야 함. 컨벤션을 강제할 도구가 없으면 일관성이 깨질 수 있음 → ADR-0005 가 그 컨벤션을 정의.
- ⚠️ "Gateway 가 다 막아주겠지" 라는 가정으로 자원에 가드를 빼먹으면 사실상 무방비. 자원의 SecurityConfig 가 진짜 가드라는 인식이 팀 전체에 박혀야 함 → 신규 자원 컨트롤러 PR 리뷰 체크리스트에 "인가 가드 명시" 항목 추가 필요.
- ⚠️ Gateway 가 검증한 클레임 헤더의 무결성은 게이트웨이를 거쳐 들어왔다는 전제에 의존. 자원 서비스로의 직접 호출이 가능한 환경(쿠버네티스 동일 클러스터 내 측면 이동) 이라면 별도 보호 필요(InternalServiceSecretFilter 같은 사이드밴드 검증).
- 🔮 재검토 트리거:
  - 자원 서비스 수가 늘면서 인가 가드 일관성이 운영 비용으로 떠오르거나
  - 정책 변경이 동시에 여러 서비스에 걸쳐야 해서 "한 곳에서 바꾸고 싶다" 가 자주 나오면
  - 회원 vs 비회원 정도의 굵직한 가드는 게이트웨이가 들고 있어야 한다는 요구가 다시 나오면
  → BFF 패턴, Policy-as-Code (예: OPA) 도입, 또는 Hybrid 인가 ADR 로 진화.

## Lessons Learned
- Gateway 의 헤더 키 변경이라는 작은 리팩토링이 권한 필터를 통째로 무력화시켰는데도 누구도 알아차리지 못했다. 정책의 정상 동작 여부를 검증할 시그널이 코드에도 운영에도 없었다는 것이다. 권한 검증은 실패했을 때 바로 401/403 으로 드러나야 동작 여부를 확인할 수 있는데, 우리 인가 필터는 항상 "헤더 없음 → 다음 필터로" 같은 방식으로 조용히 빠져나가고 있어서 죽었는지 살았는지 알 길이 없었다. 다음에 비슷한 류의 정책을 만들 때는 **정책이 한 번이라도 통과/차단을 한 적 있는가** 를 확인할 카운터나 메트릭을 같이 깔아두는 게 좋을 것 같다.

- 인증 필터와 인가 필터가 같은 식별 헤더를 두고 합의해야 했는데, 그 합의가 분산된 String 상수 몇 개로만 표현돼 있던 게 사고의 근본 원인이었다. 어휘를 enum 또는 단일 헤더 모델 객체로 묶었다면 한쪽만 바뀌어도 컴파일 에러로 잡혔을 텐데, 분산된 String 상수로 두니까 한쪽이 바뀐 걸 다른 쪽이 모른 채로 운영됐다. 두 컴포넌트가 같은 어휘를 공유한다면 그 어휘를 한 곳에 묶고 컴파일러가 동기화를 강제하게 두는 게 가장 현실적인 방법이라는 걸 배웠다.

- Step 4 에서 만든 user/member/role 모델은 사라졌는데 그걸 반영한 헤더 이름은 코드 곳곳에 그대로 살아남아서 의미가 뒤집힌 채 운영되었다. 같은 코드를 보는 사람마다 헤더 이름의 의미를 다르게 가정하게 됐고, 그 잔재가 이번 무력화 사고의 환경을 만들었다. 데이터 모델 변경 PR 에는 인터페이스 어휘 정리도 같이 들어가야 한다는 걸 다시 확인할 수 있었다.

- Step 5 에서 IdP 분리로 옛 JWT 검증기 클래스가 데드 코드가 됐는데, 만든 사람과 인수받은 사람 사이의 책임 경계가 명확하지 않아서 양쪽 다 손대지 않은 채 살아남았다. 결국 그 부산물로 남아있던 레거시 식별 헤더가 이번 무력화 사고의 직접 원인이 됐다. 인수인계 시 "deprecated 코드 목록 + EOL 일정 + 책임자" 를 명시적으로 넘겨받는 컨벤션이 있었다면 데드코드가 운영 사고로 이어지진 않았을 것이다. 조금 더 주의해야겠다.

- Step 5 의 IdP 분리, Step 6 의 서비스 분류 헤더 라우팅 도입 모두 본인이 직접 설계 주체가 아니었지만, 그 결정의 결과가 코드와 운영에 누적되어 이번 무력화 사고의 환경을 만들었다. 이런 류의 — 본인이 동의해서 한 결정이 아니라 받아들이게 된 결정 — 도 ADR 로 기록해두지 않으면, 나중에 사고가 났을 때 왜 이런 구조가 됐는지 후행 분석이 어려워진다. 좋아서 한 결정이든 아니든 똑같이 기록하는 게 ADR 의 정직함이라고 본다.

- 직관적으로는 "Gateway 가 인가까지 들고 있는 게 깔끔하다" 였지만, 막상 권한 모델이 진화하면서 깨졌다. 권한은 자원의 의미와 함께 변하는데, 그걸 자원 밖에 두면 자원이 자기 보호를 직접 정의할 수 없게 된다. 결과적으로 자원 옆에 두는 게 정합성도 운영성도 더 자연스러웠다.

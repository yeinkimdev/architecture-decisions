# ADR-0006: 운영자 인증 경계의 의도적 분리 — LDAP-only, IDP 와 분리

- Status: Proposed
- Date: 2026-05-19
- Deciders: 김예인
- Tags: #authentication #ldap #backoffice #boundary #internal-secret

## Context
- 사내에서 여러 SaaS 플랫폼을 한 콘솔에서 관리하는 통합 백오피스가 운영됨.
- 백오피스 운영자는 사내 LDAP 디렉터리에 등록된 임직원. 일반 회원과는 인증 출처가 다름.
- 일반 회원은 IDP 가 OAuth2 SSO 로 인증함.
- 운영자도 같은 IDP 로 인증할 수 있었지만 의도적으로 분리했음. 이번 ADR 은 그 결정의 배경과 결과를 명시한다.
- 이 분리는 ADR-0004 (인증 인가 책임 경계 재정의) / ADR-0005 (PlatformRole 단일 어휘) 와 연결되는 마지막 퍼즐임. 운영자 권한이 IDP 토큰에 들어가지 않는다는 점이 ADR-0005 의 PlatformRole 어휘를 "구독" 으로 깔끔하게 단일화할 수 있게 해줬음.

## Journey

### Step 1 — 운영자도 회원 DB 에 두려던 시도 (실패)
- ADR-0004 Journey Step 4 / ADR-0005 Journey Step 1 과 같은 시기.
- 처음에는 운영자도 회원 DB 의 한 행으로 표현하려 했음. 이유는 audit (`createdBy` / `modifiedBy`) 의 외래 키 정합성 — 운영자가 데이터를 수정할 때도 같은 회원 테이블에 한 행이 있어야 외래 키가 깔끔하게 박힘.
- 모델: 외부 사용자(`user`, role=USER) vs 내부 운영자(`member`, role=ADMIN). 한 테이블에 두 종류, role 로만 구분.
- 그러나 곧 어색해짐:
  - 운영자의 인증 출처는 사내 LDAP 디렉터리, 회원의 인증 출처는 자체 회원가입. 두 종류 사용자가 같은 테이블에 있으면 패스워드 컬럼 / 가입 동의 컬럼 같은 게 한쪽에만 채워지는 모델이 됨.
  - 운영자 권한과 회원 권한을 같은 role 테이블로 표현하려니 어휘가 충돌함. `ROLE_ADMIN` 이 "이 사람은 운영자" 를 의미하는지 "이 사람은 A 플랫폼의 관리자 회원" 을 의미하는지 모호.
- 이 어색함이 누적되면서 백오피스를 회원 DB 에서 분리하는 게 자연스럽다는 판단으로 이어짐.

### Step 2 — IDP 분리 + 통합 운영자 요구가 동시에 온 시점
- ADR-0004 Journey Step 5/6 와 같은 시기.
- IDP 가 OAuth2 인가 서버로 분리되면서 IDP 의 책임이 외부 고객 인증 / 토큰 발급으로 명확해졌음.
- 동시에 제품 결정으로 "백오피스는 모든 플랫폼을 한 콘솔에서 관리" 라는 통합 운영자 요구가 들어옴. 즉 한 운영자가 여러 SaaS 를 다 본다.
- 이 두 요구의 교집합이 자연스럽게 분리를 가리켰음:
  - 운영자는 IDP 를 거치지 말아야 함. IDP 는 외부 고객용 토큰 발급기인데, 운영자에게 그런 토큰을 발급하면 "어느 SaaS 의 회원인지" 어휘가 운영자에게도 적용돼서 어휘가 흐려짐 (운영자 = 모든 SaaS 의 회원? 어색함).
  - 운영자는 LDAP 으로 인증. 사내 임직원이라는 사실 자체가 인증 근거.
  - 백오피스는 자체 LDAP 인증 흐름을 가진 별도 서비스로 만들어짐. 회원과 운영자의 인증 경계가 완전히 분리됨.

### Step 3 — 사이드밴드 신뢰 도입 (공유 시크릿)
- 통합 백오피스가 모든 플랫폼의 자원 서비스를 호출해야 함. 예: 백오피스 → 자원 서비스 (예: 한 플랫폼의 게시판 도메인 서비스) 의 게시글 답글 작성 API.
- 이 호출은 IDP 토큰을 가지고 있지 않음 (운영자가 IDP 인증을 안 받으니까). 그러므로 Gateway 의 일반 인증 흐름을 못 거침.
- 자원 서비스에 별도 진입 path (예: `/backoffice/**`, `/internal/**`) 를 두고 공유 시크릿 헤더로 백오피스 호출만 신뢰하도록 했음. 시크릿이 곧 "신뢰된 운영자 호출이다" 라는 증거 역할.
- 이 흐름이 ADR-0004 의 자원 인가 위임 결정과 자연스럽게 결합됨. 자원 서비스의 SecurityConfig 가 두 가지 진입 경로(IDP 인증된 회원 / 시크릿 통과한 운영자) 를 동시에 표현하는 형태가 됨.

### Step 4 — Operator-Id 헤더 도입 (audit)
- 운영자가 자원 서비스의 데이터를 변경할 때 (예: 회원 게시판 답글 작성), audit 트레이스가 필요. createdBy / modifiedBy 또는 도메인 특화 컬럼(예: 게시판 테이블의 운영자 식별자 컬럼) 에 운영자 식별자를 박아야 함.
- 백오피스가 LDAP 인증된 운영자의 ID 를 추출해서 운영자 식별 헤더로 자원 서비스에 전달함.
- 자원 서비스는 시크릿 통과한 호출에서만 그 식별자를 신뢰함 (시크릿이 신뢰 경계, 운영자 식별 헤더는 그 안에서의 식별자).
- 이로써 Step 1 의 "audit 위해 운영자도 회원 DB 에 둬야 한다" 는 부담을 헤더 기반 트레이스로 우회할 수 있게 됨. 운영자는 회원 DB 에 없어도 audit 가 가능해짐.

## Decision
1. 운영자 인증은 LDAP 단독. IDP 와 분리한다. 운영자에게 IDP 토큰을 발급하지 않는다.
2. 백오피스는 자체 SecurityFilterChain 으로 LDAP 인증을 수행한다. Spring Security 의 form login + LDAP `AuthenticationProvider`. 세션 기반.
3. 백오피스가 자원 서비스를 호출할 때는 공유 시크릿 헤더로 신뢰 경계를 표현한다. 자원 서비스의 백오피스 진입 path 는 IDP 토큰 검증 대신 시크릿 검증을 사용한다.
4. 운영자 식별자는 별도 헤더로 자원 서비스에 전달한다. audit 트레이스에 사용. 시크릿 통과한 호출에서만 신뢰한다.
5. 이 분리는 의도적인 것으로, 운영자와 일반 회원이 다른 도메인의 사용자(임직원 vs 고객) 라서 인증 경계도 다른 차원이 자연스럽다는 도메인 판단에 기반한다.

## Alternatives Considered
- **A안: 운영자도 IDP 로 인증 (단일 인증 경로)**
  - 장점: 인증 흐름 한 가지. 외부 도구 통합(SSO, MFA 솔루션 등) 시 운영자도 같이 적용 가능.
  - 단점: IDP 가 두 가지 사용자 모델(외부 고객 / 내부 임직원) 을 동시에 들어야 함. IDP 가 LDAP 도 알아야 함. IDP 책임 비대. 운영자 권한과 회원 권한이 같은 토큰에서 표현되면 ADR-0005 의 PlatformRole 어휘에 "운영자 = 모든 플랫폼 ROLE" 같은 특별 케이스를 넣어야 해서 어휘가 흐려짐.
  - 실제로 시도되었다가 (Journey Step 1) 어색해서 분리로 회귀.
- **B안 (선택됨): 분리된 인증 경계 + 사이드밴드 신뢰 (시크릿)**
  - 장점: 각 인증 경로가 자기 도메인(외부 고객 / 내부 운영자) 에 집중. 책임 명확. ADR-0005 의 PlatformRole 어휘가 "구독" 으로 깔끔하게 단일화 가능.
  - 단점: 두 인증 경로 + 사이드밴드(시크릿) 까지 세 가지 경계 관리. 시크릿 회전 정책, 환경별 시크릿 동기화 등 운영 부담.
- **C안: mTLS 기반 서비스 ID 신뢰 (시크릿 대신)**
  - 백오피스 → 자원 서비스 호출을 클라이언트 인증서(mTLS) 로 신뢰.
  - 장점: 시크릿 회전 / 노출 위험에서 자유로움. 인프라 레벨 신뢰.
  - 단점: 인증서 발급/관리 인프라 (PKI / SPIFFE) 필요. 현재 단계에서는 운영 부담이 시크릿보다 큼. 미래 옵션으로 열어둠.

## Consequences
- ✅ IDP 가 외부 고객 인증에만 집중. 책임 명확.
- ✅ 백오피스가 모든 플랫폼을 LDAP 로 통합 관리할 수 있음 — 통합 운영자 요구 충족.
- ✅ 운영자 권한과 회원 권한이 같은 토큰에 섞이지 않음. 그 결과 ADR-0005 의 PlatformRole 어휘를 "구독" 으로 깔끔하게 단일화할 수 있었음.
- ✅ 운영자 audit 가 운영자 식별 헤더로 명시적으로 흘러감. 누가 데이터를 바꿨는지 추적 가능. 운영자가 회원 DB 에 없어도 됨.
- ⚠️ 시크릿 관리 부담이 생김. 환경별 시크릿 동기화, 회전 정책, 노출 위험. 시크릿이 유출되면 신뢰 경계가 그대로 무너짐.
- ⚠️ 통합 백오피스의 가용성이 모든 플랫폼 운영의 단일 장애점이 됨. 백오피스 장애 시 모든 플랫폼 운영 불가 (운영자만 영향. 일반 회원 서비스는 무관).
- ⚠️ 자원 서비스의 SecurityConfig 가 두 가지 진입 경로 (IDP 인증된 회원 / 시크릿 통과한 운영자) 를 동시에 표현해야 함 — 복잡도 증가. 백오피스 진입 path 를 `permitAll()` 로 두고 시크릿 검증 필터가 진짜 가드를 하는 형태.
- ⚠️ LDAP 단일 인증이라 LDAP 장애 시 운영자 모두 로그인 불가. 백오피스 장애가 곧 운영 마비.
- 🔮 재검토 트리거:
  - 백오피스가 외부(사외 망) 에서 접근 가능해져야 한다면 LDAP 만으로 부족함 (LDAP 은 사내 망 전제) → MFA / VPN / 다른 인증 메커니즘 검토.
  - 시크릿 회전이 무거워지면 mTLS / SPIFFE 같은 서비스 ID 기반 신뢰로 대체 검토 (C안).
  - 운영자 권한이 세분화되어야 한다면 (예: 플랫폼별 운영자 권한 분리, 슈퍼 운영자 vs 일반 운영자) 시크릿 단일 모델로 부족 → 운영자 권한 모델 ADR.

## Lessons Learned
- 인증 경로를 단일로 묶고 싶은 직관이 처음에는 깔끔해 보였지만, 막상 시도해보니 외부 고객과 내부 임직원이 다른 도메인의 사용자라는 점을 무시하고 있어서 곧 어색해졌음. 인증 출처(자체 회원가입 vs 사내 디렉터리) 가 다르고 권한 어휘도 다른 두 종류 사용자를 한 흐름에 묶으려니 의미 없는 컬럼이 한쪽에만 채워지거나 어휘가 충돌하는 식의 문제가 누적됐음. 사용자 모델이 두 도메인이라면 인증 경계도 두 개로 두는 게 자연스럽다는 걸 시행착오로 배웠음.

- 운영자 인증을 IDP 에서 분리한 결과, ADR-0005 의 권한 어휘가 "구독" 만으로 단일화될 수 있었음. 운영자 권한이 IDP 토큰에 안 들어가니까 PlatformRole 에 "운영자" 같은 특별 케이스를 넣을 필요가 없었음. 결정 하나가 다음 결정의 단순함을 가능하게 한 사례였고, 그 덕분에 ADR-0004 / 0005 / 0006 이 서로 자연스럽게 연결됨.

- 사이드밴드 신뢰(공유 시크릿) 는 작은 운영 부담으로 큰 보안 경계를 만들어줘서 현재 단계에서는 가장 실용적인 답이었지만, 시크릿이 유출되면 그 경계가 그대로 무너진다는 한계도 같이 있음. 영구 답이 아니라 trade-off 결정이고, 미래에 mTLS / SPIFFE 같은 서비스 ID 기반 신뢰로 대체할 가능성을 열어둘 필요가 있음.

- audit 정합성을 위해 회원 DB 구조를 흩뜨리고 싶은 유혹이 처음에 있었지만, 운영자를 회원 테이블에 두려던 시도가 곧 어색해졌음. audit 는 헤더 기반 트레이스로도 충분히 풀리고, 그 쪽이 도메인 모델 정합성을 더 잘 지키는 방식이었음. "데이터 정합성" 같은 인프라 요구를 도메인 모델에 반영하기 전에 우회 방법을 한번 고민해볼 가치가 있음.
